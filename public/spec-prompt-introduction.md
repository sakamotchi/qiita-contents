# はじめに

[仕様駆動開発](https://zenn.dev/sakamotchi/articles/ai-spec-driven-development)を実践していると、開発中の基本動作は以下の2つに集約されます。

1. **仕様ドキュメント（Markdown）を読む**
2. **ターミナルでAI CLIに指示を出す**

この2つをやるだけならVS Codeは重すぎる。メモリ1GBを使うIDEのエディタ機能やデバッガはほぼ使いません。コーディングやドキュメント作成などの作業をするのはAIの仕事だからです。

そこで、Markdownプレビューと統合ターミナルだけを備えた軽量デスクトップアプリ「**SDDesk**」を作りました。

https://github.com/sakamotchi/sddesk

:::note info
この記事は [Zenn 版](https://zenn.dev/sakamotchi/articles/spec-prompt-introduction)を元に、公開後に追加した機能（**Claude Code 通知**、**プロンプト編集パレット**、**ターミナルタブの強化**、**画像ビューア**）を追記した Qiita 版です。
:::

# 何ができるアプリか

SDDeskの画面は3つのペインで構成されています。

| ペイン | 役割 |
|--------|------|
| ファイルツリー | プロジェクト内のファイルを階層表示。ファイルの追加・削除・リネームも可能 |
| コンテンツエリア | Markdownプレビューまたはコードビューアを表示 |
| ターミナル | フルPTY対応の統合ターミナル。Claude Codeなどをそのまま実行可能 |

コンテンツエリアとターミナルは `Ctrl+Tab` で瞬時に切り替えられます。分割表示にも対応しているため、仕様を見ながらターミナルで作業する、という使い方がそのままできます。

# メモリ使用量200MB未満

SDDeskは**Tauri v2**で構築しています。ElectronではなくTauriを選んだ最大の理由はメモリ効率です。

| アプリ | メモリ使用量（目安） |
|--------|---------------------|
| VS Code | 約1GB |
| SDDesk | 200MB未満 |

Tauriはフロントエンドの描画にOS標準のWebViewを使うため、ChromiumをバンドルするElectronと比べてメモリ消費が大幅に少なくなります。

仕様駆動開発ではAI CLIが常駐するため、IDE側のリソース消費を抑えられることは実用上の大きなメリットです。

# 主な機能

## Markdownプレビュー

仕様ドキュメントの閲覧に特化したプレビュー機能です。

- **Mermaidダイアグラム対応**: 設計書に含まれるフローチャートやシーケンス図をそのまま描画
- **ライブ更新**: ファイルの変更を検知して自動でプレビューを更新
- **GFM対応**: テーブル、タスクリストなどGitHub Flavored Markdownをサポート

## コードビューア（読み取り専用）

15以上の言語に対応したシンタックスハイライト付きのコードビューアです。TypeScript、Python、Rust、Goなど主要な言語をカバーしています。

ここで重要なのは、**エディタではなくビューア**である点です。仕様駆動開発ではコードの編集はAIに任せるため、人間はコードを「読む」ことに集中します。この割り切りがアプリの軽量さにも寄与しています。

## 画像ビューア

PNG / JPEG / GIF / WebP / BMP / ICO / AVIF / SVG に対応した画像ビューアを備えています。設計書に添付された UI モックやスクリーンショットもそのまま表示できます。

## 統合ターミナル

Alacrittyのターミナルエンジン（Rust実装）をCanvas 2Dでレンダリングする独自のターミナルエミュレータを搭載しています。

- **フルPTY対応**: zsh/bashのフル機能が使える本格的なターミナル
- **複数タブ**: `Cmd+T` で新しいターミナルタブを追加
- **分割表示**: ターミナルペインの分割にも対応

Claude Codeのような対話型AIツールもそのまま動作します。

## Cmd+Click / パスパレットでパスをターミナルに挿入

AI CLIにファイルパスを指定する場面は非常に多いです。SDDeskでは2つの方法でパスをターミナルに直接挿入できます。

**Cmd+Click**: ファイルツリー上のファイルを `Cmd+Click` すると、そのパスがアクティブなターミナルに即座に挿入されます。仕様書を見ながら「このファイルを修正して」と指示するとき、パスを手打ちする必要がありません。

**パスパレット（Ctrl+P）**: プロジェクト内のファイルをファジー検索し、選択したファイルのパスをターミナルに挿入します。ファイル数が多いプロジェクトではこちらが便利です。

どちらも地味な機能ですが、AI CLIとの対話では繰り返し使うため、作業効率に直結します。

## ドキュメントステータス管理

MarkdownのYAMLフロントマターに `status` フィールドを記述すると、ファイルツリー上にステータスバッジが表示されます。

```yaml
---
status: draft      # draft | reviewing | approved
---
```

仕様書のレビューフローを視覚的に管理できます。

# 公開後に追加した機能

Zenn 版の公開以降、実際に使い込む中で「ここが欲しい」と感じた機能をいくつか追加しました。

## Claude Code 通知（OS ネイティブ通知）

Claude Code に長めのタスクを任せて別の作業をしていると、「Permission 待ちになっていたのに気づかなかった」ということが起きます。そこで、アプリ外でも処理状況を OS ネイティブ通知で受け取れるようにしました。

- **OSC 9 検出**: PTY 出力中の `ESC ] 9 ; <message> BEL` を検知して通知を発火
- **HTTP フックサーバ**: `127.0.0.1:19823` で Claude Code の hook を受信（ループバックのみで待ち受け）
- **通知種別の自動分類**: メッセージ本文から **Permission / Completed / Error / Waiting / Attention** を判定
- **タブ識別**: 通知タイトルを `Claude Code — {タブ名}` 形式にして、複数ターミナルでもどのタブからの通知か分かる
- **フォーカス判定**: アプリがフォーカス中かつ発火元タブがアクティブなときは通知を抑止
- **未読マーク連携**: 通知が出たタブには琥珀色のドットと左ボーダーが付き、タブをアクティブ化すると自動クリア

設定画面から ON / OFF を切り替えられます（デフォルト ON）。

## プロンプト編集パレット（`Cmd+Shift+P`）

Claude Code に複数行のプロンプトを送るとき、ターミナルに直接打ち込むと「途中で Enter を押して誤送信してしまう」「IME の確定 Enter で送ってしまう」といった事故が起きがちです。これを解消する**誤送信しないための推敲 UI** として、プロンプト編集パレットを追加しました。

- **起動**: `Cmd+Shift+P`（Win/Linux は `Ctrl+Shift+P`）、もしくはターミナルタブ／本体の右クリックメニューから
- **複数行編集**: 初期 8 行の textarea、縦リサイズ可。`Enter` は改行、`Cmd+Enter` で送信
- **非モーダル**: パレット表示中もツリーやターミナルをそのまま操作できる（VS Code のコマンドパレット的な UX）
- **パス挿入ディスパッチ**: パレット開時は `Cmd+Click` / 右クリック挿入 / `Ctrl+P` のパス確定が **textarea のキャレット位置に挿入**される。閉時は従来どおりターミナルに直接挿入
- **下書き保持**: タブごとに下書きをメモリに保持。パレットを閉じて再度開くと復元される
- **IME 二重ガード**: `compositionstart/end` と `isComposing` の両方で変換中の Enter / Cmd+Enter を抑止
- **タブ閉鎖で自動破棄**: ターミナルタブを閉じると該当下書きをクリア。パレットの送信先だった場合はパレット自体も閉じる

「書き終わるまでは絶対に送信されない」状態を作れるので、Claude Code への長文プロンプトや手順書のコピペが安心して行えます。

## ターミナルタブの強化

- **動的タブタイトル（OSC 0/1/2）**: シェルの `PROMPT_COMMAND` や `precmd` で出力するタイトルをタブ名に反映
- **手動リネーム**: タブをダブルクリックでインライン編集。確定するとピン留めされ、OSC 更新を無視
- **表示タイトル優先順位**: `pinned && manualTitle` > `oscTitle` > `fallbackTitle`
- **シェル終了で自動クローズ**: PTY が exit するとタブを自動で閉じる（最後の 1 枚は新しい空タブを生成）
- **`Shift+Enter` で LF 送信**: 通常 Enter は `\r`（CR）のまま、Shift+Enter は `\n`（LF）を送る。Claude Code のように「Enter は送信、Shift+Enter は改行」を期待するツールに対応

## ターミナル本体の右クリックメニュー

これまでタブ側の右クリックしか存在しませんでしたが、ターミナル描画領域自体にも右クリックメニューを追加し、そこからプロンプト編集パレットを起動できるようにしました。

# キーボードショートカット

SDDeskはキーボード操作を重視しています。主なショートカットは以下の通りです。

| ショートカット | 操作 |
|---------------|------|
| `Ctrl+Tab` | コンテンツ / ターミナルの切り替え |
| `Ctrl+P` | パスパレットを開く |
| `Cmd+Shift+P` / `Ctrl+Shift+P` | プロンプト編集パレットを開く |
| `Cmd+Enter`（パレット内） | プロンプト編集パレットから送信 |
| `Cmd+T` | 新しいターミナルタブを作成 |
| `Cmd+W` | アクティブなタブを閉じる |
| `Cmd+\` | コンテンツの分割表示を切り替え |
| `Cmd+Shift+\` | ターミナルの分割表示を切り替え |
| `Cmd+0` | ファイルツリーにフォーカス |
| `Cmd+1-9` | n番目のタブに移動 |
| `Shift+Enter`（ターミナル） | LF を送信（改行のみで実行しない） |
| `?` | ショートカット一覧を表示 |

## なぜxterm.jsではなくAlacrittyか

ターミナルレンダラは当初xterm.jsを使っていましたが、以下の理由からAlacrittyベースのCanvas 2Dレンダラに移行しました。

- **CJK文字の描画問題**: xterm.jsでは日本語などのCJK文字が文字化けするケースがあった
- **描画パフォーマンス**: DOM操作を介さずCanvas 2Dに直接描画するため高速

Rustで実装されたAlacrittyのターミナルエンジンがセル単位の描画をCanvas 2Dに出力する構成です。

# 典型的なワークフロー

1. **左ペイン**: ファイルツリーから仕様ドキュメントを選択
2. **右上ペイン**: Markdownプレビューで仕様を確認
3. **右下ペイン**: ターミナルでClaude Codeに実装を指示
4. 長めのプロンプトは `Cmd+Shift+P` で**プロンプト編集パレット**を開き、ツリーから `Cmd+Click` でパスを挿入しながら推敲して `Cmd+Enter` で送信
5. 長時間処理は **Claude Code 通知**に任せ、他のタブで別作業。通知が来たらタブの琥珀マークで気づける

仕様書自体の作成・編集もClaude Codeに指示します。人間が仕様の方針を決め、ドキュメントへの落とし込みはAIに任せる。Markdownプレビューで仕上がりを確認し、修正があればターミナルから追加の指示を出す。この一連の流れがSDDeskの中で完結します。

# インストール

## macOS

[GitHubリリースページ](https://github.com/sakamotchi/sddesk/releases)から `.dmg` ファイルをダウンロードしてインストールできます。

## ソースからビルド

```bash
git clone https://github.com/sakamotchi/sddesk.git
cd sddesk
npm install
npm run tauri build
```

Node.js 20以上とRust（stable）が必要です。

# おわりに

仕様駆動開発を続ける中で、「仕様を読んでAIに指示する」という作業に対してVS Codeはオーバースペックだと感じるようになりました。必要なのはMarkdownプレビューとターミナルだけです。

SDDeskはその最小限の機能に絞ることで、メモリ200MB未満という軽量さを実現しています。コードエディタを載せないという割り切りは、「コードを書くのはAIの仕事」という仕様駆動開発の思想そのものです。

公開後に追加した **Claude Code 通知** と **プロンプト編集パレット** は、どちらも「AI CLI とのやり取り」という核の体験を直接良くするための機能です。軽量性を保ったまま、仕様駆動開発の現場で本当に効く改善を積み上げていこうと思います。

まだ開発途中で、改善の余地は多くあります。フィードバックや要望はGitHubのIssueでお待ちしています。
